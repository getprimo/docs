Windows Autopilot gives new Windows devices an out-of-the-box setup experience. When an employee powers on a device for the first time, they sign in with their Microsoft credentials and all company apps, settings, and security policies are applied automatically — no IT intervention required.

This article covers the full setup: from registering the FleetDM domain on Azure to creating deployment profiles and testing the Autopilot experience.

## Prerequisites

- A Microsoft account with administrator access
- An Entra "tenant"
- A license that includes **Microsoft Entra ID Plan 1** (or **Plan 2**) & **Windows Autopilot**

The minimum license covering both features is **Enterprise Mobility + Security E3** ([compare plans](https://www.microsoft.com/fr-fr/microsoft-365/enterprise-mobility-security/compare-plans-and-pricing)).

## Employee experience on first startup

1. Power on the device
2. Select country and language, configure keyboard layout
3. Connect to the internet
4. Accept FleetDM terms and conditions
    ![Screenshot: What happens when an employee arrives with a workstation configured with Autopilot?](/images/notion-import/zero-touch-enrollment-for-windows__img-025__block-0018__1b243d9f.png)

5. Sign in with Microsoft credentials (+ 2FA if enabled)
    ![Screenshot: What happens when an employee arrives with a workstation configured with Autopilot?](/images/notion-import/zero-touch-enrollment-for-windows__img-026__block-0020__1b243d9f.png)

6. Set up a PIN code and fingerprint (if supported)
    ![Screenshot: What happens when an employee arrives with a workstation configured with Autopilot?](/images/notion-import/zero-touch-enrollment-for-windows__img-027__block-0022__1b243d9f.png)

7. The workstation is ready — apps and policies defined in the Primo Cockpit are applied automatically

## Restrictions

Autopilot devices use **Entra accounts** rather than local accounts. Microsoft is progressively deprecating local accounts in favor of Entra accounts.

The Entra administrator can configure whether users have standard or administrator account privileges. Entra users with the **Global Administrator** role are automatically administrators on all Autopilot machines they sign into.

Internet access is required during the first boot to authenticate with Entra.

## How to set it up

The setup assumes your company is called ACME and uses the domain `acme.com`.

## Step 1 — Register the FleetDM domain and create the Fleet application on Azure

<Note>
This step requires coordination with Primo support. When you register the FleetDM domain in Azure, Microsoft generates a verification code that must be communicated to the Primo team. Allow up to **2 business days** for the DNS configuration to propagate before continuing.
</Note>

You will perform three operations in the Microsoft Azure console with administrator rights:
1. Declare the FleetDM domain as legitimate with Azure
2. Create the Fleet application and grant it MDM permissions
3. Set FleetDM as the default MDM for new devices (replacing Intune if applicable)

### 1. Declare the FleetDM domain

- Sign in with an administrator account at: [Microsoft Azure](https://portal.azure.com/)
- Search for and click on [**Domain names**](https://portal.azure.com/#view/Microsoft_AAD_IAM/DomainsList.ReactView)
- Click on **+ Add custom domain**
- In the field, enter **{company}.mdm.getprimo.com** (i.e. [**acme.mdm.getprimo.com**](http://acme.mdm.getprimo.com/), [ask support](mailto:support@getprimo.com?subject=Connaître%20le%20domaine%20de%20notre%20instance%20FleetDM) ([support@getprimo.com](mailto:support@getprimo.com)) if you don't know this domain name)
- Share with us the value of the **Destination or routing address** field (in the format `MS=ms12345678`)
- Wait for our response (2 business days at most) before continuing with the procedure
- You can then click on **Verify**

### 2. Create the FleetDM application

- Sign in with an administrator account at: [Microsoft Azure](https://portal.azure.com/)
- Search for [**Mobility (MDM and MAM)**](https://portal.azure.com/#view/Microsoft_AAD_IAM/MdmAppsListBlade)
- Choose **+ Add application**, then choose **+ Create your own application**
- Enter **Fleet** as the application name and click **Create**
- Fill in
  - MDM terms of use URL: `https://{company}.mdm.getprimo.com/api/mdm/microsoft/tos`
  - MDM discovery URL: `https://{company}.mdm.getprimo.com/api/mdm/microsoft/discovery`
- Click **Save**
- Return to [**Mobility (MDM and MAM)**](https://portal.azure.com/#view/Microsoft_AAD_IAM/MdmAppsListBlade)
- Click on the **Fleet** application then on **Custom MDM application settings**
- Click on the link below **Application ID URI** then click **Edit**
- Enter your **Fleet** instance address (`https://{company}.mdm.getprimo.com`) and click **Save**
    ![Screenshot: 2. Creating the FleetDM application](/images/notion-import/zero-touch-enrollment-for-windows__img-028__block-0065__27743d9f.png)

- Choose **API permissions** then **Add a permission**
- Click on **Microsoft Graph** then on **Delegated permissions**, then select:
  - **Group > Group.Read.All**
  - **Group > Group.ReadWrite.All**
  - and click **Add permissions**
      ![Screenshot: 2. Creating the FleetDM application](/images/notion-import/zero-touch-enrollment-for-windows__img-029__block-0079__27743d9f.png)

- Then return to **API permissions** and **Add a permission**, and choose **Microsoft Graph** again
- This time, click on **Application permissions**, and add the following permissions:
  - **Device > Device.Read.All**
  - **Device > Device.ReadWrite.All**
  - **Directory > Directory.Read.All**
  - **Group > Group.Read.All**
  - **User > User.Read.All**
  - and click **Add permissions**
- Once back on the API permissions screen, click on **Grant admin consent for ACME**

The Fleet application is now declared as a legitimate MDM with the Azure portal.

### 3. Set FleetDM as the default MDM for new devices

- Sign in with an administrator account at: [Microsoft Azure](https://portal.azure.com/)
- Go to [**Mobility (MDM and MAM)**](https://portal.azure.com/#view/Microsoft_AAD_IAM/MdmAppsListBlade)
- Click on **Microsoft Intune**
- In **MDM user scope**, select **None**
- In **MAM user scope**, select **None**
- Click **Save**
- Go to [**Mobility (MDM and MAM)**](https://portal.azure.com/#view/Microsoft_AAD_IAM/MdmAppsListBlade)
- Click on **Fleet**
- In **MDM user scope**, select **All**
- In **MAM user scope**, select **All**
- Click **Save**

The Fleet application is now declared as the MDM that will handle new devices with the Azure portal.

**Note**: If you have started enrolling devices on Primo/FleetDM before performing these operations, please let us know so we can force the MDM change from Intune to FleetDM by running a script (without which the machine is in an inconsistent state that may affect the Primo experience).

## Step 2 — Create Autopilot deployment profiles

To complete the Autopilot setup, it is necessary to create at least one "deployment profile".

You can find Microsoft's official documentation on Deployment Profiles at this link: [https://learn.microsoft.com/fr-fr/autopilot/profiles](https://learn.microsoft.com/fr-fr/autopilot/profiles).

Through this profile, it is possible to define the following settings:

- Choice of user type: **Administrator** or **Standard**

Note: Entra users with a **Global Administrator** role will be administrators of all machines regardless of this setting's value

- Default country choice
- Default keyboard language choice
- Machine naming pattern

For example: `ACME-%RAND:5%` will generate the following names:
- ACME-23456
- ACME-98479
- ACME-19838

To create a deployment profile, follow these instructions:

- Go to: [Microsoft Intune admin center](https://intune.microsoft.com/#view/Microsoft_Intune_Enrollment/AutopilotDeploymentProfiles.ReactView)
  - or sign in to [https://intune.microsoft.com/](https://intune.microsoft.com/) and go to **Devices** > **Enrollment** > **Deployment Profiles**
- Click **+ Create profile**
- The first step allows you to name the profile
- The second allows you to describe the deployment modes
- The third step allows you to assign profiles to devices

Create a dynamic group to target all ZTD devices:

- Go to: [portal.azure.com](https://portal.azure.com/#view/Microsoft_AAD_IAM/GroupsManagementMenuBlade/~/Overview)
- Click **+ Create group**
- Name it "Windows Autopilot devices", for example
- As target select **"dynamic devices"**
- For the dynamic rule, use **(device.devicePhysicalIds -any (_ -startsWith "[ZTDid]"))**
- Click **Create**

Finally, create an Enrollment Status Page configuration:

- Go to: [Microsoft Intune admin center](https://intune.microsoft.com/#view/Microsoft_Intune_Enrollment/EnrollmentStatusPageProfileList.ReactView)
  - or sign in to [https://intune.microsoft.com/](https://intune.microsoft.com/) and go to **Devices > Enrollment > Enrollment Status Page**
- Click **+ Create**
- The first step allows you to name the profile (i.e. Primo)
- The second allows you to define whether you want to show the progress to the user or not. We suggest "yes". If yes, you can put the settings you want. You can keep the default values
- The third step allows you to assign the ESP to devices. Assign it to your dynamic group.

## Step 3 — Test the Autopilot experience

### Sharing Intune admin console access with InMac

For Autopilot to work automatically on devices ordered through Primo, it is necessary to perform an operation with our partner reseller InMac:

- Establish a Microsoft partnership relationship between InMac and you

This access will allow them, for each ordered device, to inform Intune of its existence and ensure that it is handled by Primo during its initialization.

To do this, you must use the following link from an account with the **Global Administrator** role:

- [admin.microsoft.com](https://admin.microsoft.com/Adminportal/Home?invType=ResellerRelationship&partnerId=576fd34d-ccdb-48fb-a65e-ece0e985cd46&msppId=0#/partners/invitation)

Follow the instructions until accepting the partnership relationship.

### Alternative: testing on existing hardware (requires device reset)

There is another way to test the Autopilot experience without necessarily ordering new devices through Primo (and therefore without having to establish the partnership relationship and granular access to the admin console described in the previous paragraph).

To do this, simply follow [these instructions from our partner FleetDM](https://fleetdm.com/guides/windows-mdm-setup#step-2-register-a-test-workstation) (*Step 2: register a test workstation*) which describes the process that consists of:

- extracting the device hash (its serial number, so to speak)
- importing it into the list of Autopilot devices on your Azure/Intune console (this is how the link is made between this device and your company)
- resetting the device

The user can then enjoy the Autopilot experience.
